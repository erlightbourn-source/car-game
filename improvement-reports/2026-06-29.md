## Research findings (genre / competitor)

1. **Adjustable game speed is the proven accessibility lever for young kids.** Snoot Runner's
   reviewers single out its adjustable game speed as what "dramatically expands the game's
   inclusivity" for younger / less reflex-comfortable players. Common Sense Media separately
   notes ages 4-7 struggle with the "never-ending" nature of endless runners and self-regulation.
   *Implication for Lane Rush:* the existing Easy mode is the right structural answer, and it
   should be genuinely near-endless for toddlers. Today it measures correctly at 300s median /
   73% reach-5min — good. This finding is what motivated auditing the easy-mode metric (see Shipped).
   Source: https://gamecritix.co.uk/snoot-runner-review/ , https://www.commonsensemedia.org/lists/endless-runner-apps

2. **The difficulty curve should keep the player "right at the edge of their ability" — that's
   where flow happens — but ramping too fast frustrates newcomers before they can improve.**
   *Implication:* Lane Rush's `passed`-driven ramp with an early plateau + late escalation
   (HARD_AT_PASS) matches this; the cohort table confirms tweens/teens hit the 5-min cap while
   young/toddler fail earlier — a correct, gradual curve. No change warranted this run.
   Source: https://ejaw.net/what-is-an-endless-runner-game/

3. **"Aha moment" inside the first 60 seconds drives retention** (users who see value in 60s are
   3-5x more likely to stay; smooth onboarding = up to 50% better week-1 retention); favor
   interactive learn-by-doing over text, tooltips < 140 chars.
   *Implication:* Lane Rush has a first-run tutorial; a future run should audit it for the
   60-second value moment and tooltip length rather than assume it's optimal. Flagged below.
   Source: https://www.appcues.com/blog/in-app-onboarding , https://www.zigpoll.com/content/how-can-we-optimize-the-onboarding-experience-to-reduce-player-dropoff-rates-in-our-mobile-game

4. **Low-end mobile web perf: cap framerate (~30fps on weak phones), scale by devicePixelRatio,
   minimize draw calls / object churn in the loop, profile on real hardware not emulators.**
   *Implication:* Lane Rush already has an adaptive-quality watchdog, lowered base DPR, and a
   GPU pixel budget (recent commits). A future perf pass could verify the 30fps fallback and
   per-frame allocation; not low-risk enough to touch blind this run.
   Source: https://www.construct.net/en/blogs/construct-official-blog-1/boosting-mobile-html5-game-performance-with-webgl-791 , https://www.airtightinteractive.com/2015/01/building-a-60fps-webgl-game-on-mobile/

## Shipped this run

- **`c71b141` — Fix playtest harness: easy-mode toddler metric was measuring NORMAL mode.**
  `runCohort()` hardcoded `e.assist = false` at the start of every run, overwriting the
  `e.assist = true` the caller set for the easy-toddler measurement. The agent's own
  instrument was therefore misreporting the single band targets care most about for easy mode.
  - Impact: the "Easy-mode toddler" report line now reflects actual easy mode.
  - Harness before -> after: easy-toddler median **49s -> 300s**, reach-5min **0% -> 73%**.
  - Fairness oracle: 100/100% -> 100/100% (PASS, exit 0). All other bands unchanged within noise.
  - `tools/` only, not loaded by index.html, so no `?v=` bump required.

No gameplay/config changes shipped — easy mode was already correctly tuned; the apparent
"easy mode is too hard for toddlers" signal was a measurement bug, not a game bug. Per policy,
the game was not churned just to look busy.

## Flagged for human review

- **Onboarding "first 60 seconds" audit (S/M).** Research shows the first-session value moment
  drives 3-5x retention. Worth a human pass over the first-run tutorial: does the player get an
  "aha" (first near-miss combo / first coin grab) inside 60s? Are tooltips < ~140 chars and
  learn-by-doing? Touches UI copy/flow — judgment call, flag rather than auto-edit.
- **Verify the low-end-phone 30fps fallback + per-frame allocations (M).** Adaptive quality
  exists; confirming the framerate cap actually engages and the game loop avoids object churn
  needs real-device profiling, which the headless harness can't see. Risky to tune blind.
- **In-app adjustable speed slider (M).** Snoot-Runner-style continuous speed control would beat
  a binary Easy/Classic toggle for the 4-7 cohort. This is a new control/UX surface (not a pure
  config tweak) and should be designed by a human.

## Next up (single highest-leverage)

Onboarding first-60-seconds audit — it's the best-evidenced retention lever and Lane Rush already
has the tutorial scaffolding, so the change is bounded. Pair it with a quick check that the
tutorial's "aha" beat fires early.
